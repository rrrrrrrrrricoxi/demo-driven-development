# SCENE-FIT —— DDD 场景适配自检

给人看的版本(讲道理,方便日后校准)。行为层落在 `skills/ddd-workflow/SKILL.md` 的「第 0 步:场景自检」——那边只留行为指令(问句、报一行、收敛规则),道理在这;两问的行为层定稿以 SKILL 为准。出处:首宿主项目用了一阵 DDD 之后的灵魂拷问「我们现在这种场景适用 DDD 吗」。

## 核心机制(DDD 到底在卖什么)

把一个抽象、难描述的选择,变成**便宜、自包含、可并排的实物**,让有判断力但没时间趟代码的人**几秒定夺**。

**三个同时满足才适用**:

1. **SEE-IT 而非 KNOW-IT** —— 看得见才判得准,不是想清楚就有对错;
2. **demo 成本 << 真东西成本**,且够逼真、判断能迁移;
3. **瓶颈是人的判断,不是建造**(有个有品味、不趟代码的人卡在「哪个对味」)。

## 适用 / 不适用

| 适用(SEE-IT,像素层岔路口) | 不适用(用了反成负担) |
|---|---|
| 视觉设计 / UI 布局 / 交互手感 | 正确性问题(迁移安全吗、转义了吗)→ 测试 / 类型 / 对抗审查,demo 什么都告诉不了 |
| 文案语气 / 信息架构 A/B | 答案可推导(数据模型 / 口径 / schema)→ 资深一句「显然 X」,做 demo 去「发现」X 是演戏 |
| 多个都说得通、得看气质挑 | demo 无法便宜地逼真(要真数据 / 真延迟 / 真规模,如「3000 行列表卡不卡」)→ mock 会骗你 |
| 挑错了也便宜改 | 无真岔路(即便低风险可逆)→ 立卡 + demo + 评审的仪式比直接做还贵 |

## 两条自检(第 0 步的出处)

1. 这是「看一眼」决策(SEE-IT)还是「想清楚」决策(KNOW-IT)?
2. 这里真有岔路吗,还是在给一个**已定 / 不值得定**的选择配仪式?

## 收敛规则(治串行过度迭代)

并排发散不限轮(一轮 5 个方向并排 = 机制本身);要数的是**评审后再来一轮**的串行轮数。串行第 3 轮起,每轮开工前先答:**这轮会改变选择,还是只在磨已选中的?**答不出会改变什么 → 陈述边际递减、建议就地拍板——宣布收敛也是一种交付。

## 三个实证(首宿主项目,同一时期的天然对照)

| 案例 | 判定 |
|---|---|
| 登录 5 方向选型 | 教科书级适用:发散视觉、并排一眼选定 |
| logo 串行 20 轮 | 机制适用但失败:demo 太便宜,决定了不值得决定的事——反模式命名「logo-20 轮」 |
| plugin 自身加固(项目里价值最高的活) | 正确地没用 DDD:spec → 对抗验证 |

## 与「地基先于像素」的关系

DDD 按构造服务**像素那半步**,不管地基(数据模型 / 口径 / 迁移)。用它定地基 = 拿错工具;地基由约束和正确性决定,不是看 demo 定的。

## 为什么不是 hook(否决记录,防翻案)

hook 拿到的信号(改了哪些文件、跑了什么命令)不携带「SEE-IT 还是 KNOW-IT」信息;「新 demo + 新卡」恰是正当 DDD 工作的签名,按它触发 = 误报率 ≈ 正当使用率,人会被训练成无视——**误报比不做更糟**,还连坐孤儿守卫攒下的零误报信用(现有 Stop 守卫的威信建立在确定性事实核查上,别拿去赌)。独立 advisory skill 同否:误用的人不知道自己在误用,不会主动调它;常驻 skill 列表的 description 税挣不回位置。故落点是 SKILL 第 0 步(触发点 = 准备做 DDD 的那一刻,天然零误报)+ 本文档(存疑再读)。将来若有人重提检测器,请先拿实据:能区分「一轮 5 个并排变体」与「20 轮串行磨」的信号在哪。

## 一句话

**demo 便宜 ≠ 决策值得。看得见才判得准的岔路口才配 demo;有对错的问题走 spec → 验证,已定的选择直接做。**
